# bz-lottery 代码审查技能

## 使用场景

当用户说“review”“审查”“检查改动”“提交前看一下”等，或开发流程需要对当前 diff 做质量检查时，使用本技能。

审查只读为主，不主动修改代码；如果用户要求修复，再进入开发流程。

## 工作流

1. 查看当前改动：
   ```powershell
   git status --short
   git diff --stat
   git diff
   ```
2. 判断改动类型：后端、前端、部署、文档或 Git 流程。
3. 按检查清单审查正确性、回归风险、缺失验证和安全问题。
4. 优先输出问题；没有问题时明确说明未发现阻塞项。
5. 只在必要时提出风格建议，避免把个人偏好当作问题。

## 后端检查

- 分层是否正确，Controller 是否绕过 Service。
- 请求体是否使用校验注解，分页是否有限制。
- 接口响应、错误码、鉴权和 Swagger/OpenAPI 是否同步。
- 是否存在 N+1 查询、循环内远程调用或超大 IN 查询。
- 库存、余额、抽奖记录等关键逻辑是否并发安全。
- 数据库结构变更是否使用 Flyway。
- 关键日志是否包含 `traceId` 和业务主键。

## 前端检查

- 是否使用 Vue 3 Composition API 和现有 HTTP 客户端。
- 是否处理加载态、空态、错误态。
- 路由、菜单、权限和角色守卫是否同步。
- 是否绕过网关或硬编码后端地址。
- TypeScript 类型是否清晰，是否引入未处理的 Promise。

## 部署检查

- 是否通过 `deploy/scripts/` 统一入口操作环境。
- compose 分层是否清晰，生产配置是否混入开发配置。
- 密钥和可变参数是否外置。
- 端口、卷、Nginx 模式和环境变量说明是否同步。
- compose 文件是否可通过 `docker compose config --quiet`。

## 输出格式

```markdown
## 审查结果

总体结论：PASS / FAIL

### 问题

| 严重级别 | 文件 | 问题 | 建议 |
| --- | --- | --- | --- |
| ERROR | path | ... | ... |
| WARN | path | ... | ... |

### 验证缺口

### 备注
```

## 严重级别

- `ERROR`：可能导致编译失败、运行错误、数据错误、安全问题或明显回归，必须修复。
- `WARN`：有维护性、兼容性或边界风险，建议修复。
- `INFO`：非阻塞建议，仅供参考。
