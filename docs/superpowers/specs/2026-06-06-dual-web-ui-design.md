# 抽奖平台双端 Web UI 设计

## 背景

`bz-lottery` 当前前端已经具备登录、工作区布局、抽奖页、奖品中心、运营页、个人中心等基础页面。现有页面更像单一控制台，用户参与端和运营管理端混在一起，适合作为练习项目，但不利于后续演示、接单和扩展。

本次 UI 优化目标是：在同一个 Vue 项目内规划 PC Web 运营后台和手机 Web 用户端。第一版不拆成两个前端项目，避免维护成本过高。

## 目标

1. 保留一套 `lottery-frontend` 工程。
2. 拆分两个路由入口：
   - `/admin`：PC 运营后台。
   - `/app`：手机 Web 用户端。
3. 共用登录态、API 封装、权限判断和抽奖业务能力。
4. 后台偏信息密度和运营效率，H5 偏参与感和移动端操作体验。
5. 第一版保证可演示、可构建、桌面与手机视口可用。

## 非目标

1. 不在第一版拆成 `lottery-admin` 和 `lottery-h5` 两个项目。
2. 不重写后端 API。
3. 不在第一版补全所有活动、奖品、记录的完整 CRUD。
4. 不依赖 Stitch 作为唯一设计来源。Stitch 可用于后续生成视觉参考，但实现以现有 Vue 项目为准。

## 路由设计

### 共用入口

```text
/login
```

登录页继续复用现有登录/注册逻辑。登录成功后，根据用户角色或默认入口跳转：

- 管理员默认进入 `/admin/overview`。
- 普通用户默认进入 `/app/home`。

### PC 运营后台

```text
/admin
  /overview      运营概览
  /activities    活动运营
  /prizes        奖品中心
  /records       中奖记录
  /ai            AI 分析
  /profile       个人中心
```

后台页面以运营管理为主，适合桌面宽屏：

- 概览页展示活动数、奖品库存、抽奖次数、中奖率、待审核记录等摘要。
- 活动运营页展示活动状态、时间、参与规则和操作入口。
- 奖品中心页展示奖品等级、库存、概率和配置入口。
- 中奖记录页展示最近中奖记录和状态。
- AI 分析页承接已有抽奖数据分析能力。
- 个人中心展示当前用户、角色和权限信息。

### 手机 Web 用户端

```text
/app
  /home       活动首页
  /draw       抽奖页
  /records    我的记录
  /profile    我的
```

手机端以用户参与为主：

- 首页展示当前活动、奖池摘要、剩余机会和入口按钮。
- 抽奖页突出奖池、抽奖按钮、滚动高亮和结果反馈。
- 我的记录展示中奖历史、待审核状态和最近参与时间。
- 我的页面展示昵称、角色、退出登录等基础操作。

## 布局设计

### AdminLayout

后台使用桌面管理台结构：

- 顶部栏：项目名、当前用户、退出按钮。
- 左侧导航：概览、活动、奖品、记录、AI、我的。
- 主内容区：标题、说明、页面主体。
- 小屏适配：导航收缩为顶部菜单或抽屉式菜单。

后台需要更克制的信息密度，不做营销式大屏。

### MobileAppLayout

手机端使用移动应用结构：

- 顶部栏：当前活动/页面标题。
- 主内容区：移动端卡片、抽奖交互、记录列表。
- 底部 Tab：首页、抽奖、记录、我的。

底部 Tab 固定在视口底部，按钮区域足够大，避免移动端误触。

## 视觉系统

当前样式偏米色和金色，适合作为活动点缀，但整体偏单一。新版本统一为：

- 主色：蓝绿系，用于按钮、导航激活、关键交互。
- 点缀色：金色，仅用于奖品、中奖结果和活动强调。
- 背景：浅灰白，降低长时间管理台使用疲劳。
- 状态色：成功、警告、错误、信息分别使用清晰可辨的颜色。
- 圆角：后台卡片控制在 8px 左右；H5 端可略柔和，但不做过度圆润。

视觉目标：

- PC 后台像正式 SaaS 管理系统。
- 手机 H5 像可参与的活动小程序页面。
- 两端共享品牌感，但不共享同一种信息密度。

## 组件与状态

第一版优先抽出轻量组件，不引入复杂组件库：

- `AdminLayout.vue`
- `MobileAppLayout.vue`
- `StatCard.vue`
- `EmptyState.vue`
- `LoadingState.vue`
- `ErrorState.vue`
- `PrizeCard.vue`
- `RecordItem.vue`

数据组件必须处理三态：

- loading：加载中。
- empty：暂无数据。
- error：展示错误信息。

## API 与数据流

继续使用现有 `src/api/http.js` 和 `src/api/lottery.js`。

抽奖相关流程：

```text
Mobile Draw Page
→ fetchPrizeList()
→ fetchLotteryRecords(userId)
→ drawLottery(userId, requestNo)
→ 更新奖池高亮、结果弹窗、记录列表
```

AI 分析流程：

```text
Admin AI Page
→ streamLotteryUserDataAnalysis(userId, focus)
→ 展示分析状态、指标、推理过程、结论和建议
```

## 权限策略

第一版延续现有 `sessionState`、`hasRole`、`hasMenu` 逻辑。

建议规则：

- `/admin/**`：需要 `LOTTERY_ADMIN` 或管理菜单权限。
- `/app/**`：登录用户可访问。
- `/login`：未登录访问。

如果后端权限暂未完全区分，前端先在路由元信息中表达边界，避免 UI 混乱。

## 实现计划

1. 调整路由结构，新增 `/admin` 和 `/app` 分组。
2. 新增 `AdminLayout` 和 `MobileAppLayout`。
3. 将现有工作区页面迁移为后台视角。
4. 新增手机端首页、抽奖页、记录页、我的页面。
5. 重构全局样式变量，避免单一米金主题。
6. 为关键数据区域补 loading、empty、error 状态。
7. 执行 `npm run build`。
8. 使用浏览器检查桌面和手机视口。

## 验证标准

1. `npm run build` 通过。
2. `/login` 可访问。
3. `/admin/overview` 桌面视口布局正常。
4. `/app/home`、`/app/draw` 手机视口布局正常。
5. 抽奖页在无数据、加载中、接口错误时都有合理提示。
6. 文本不溢出按钮或卡片。
7. 移动端底部 Tab 不遮挡主要操作按钮。

## 风险

1. 当前前端代码是 JavaScript，而规范写的是 TypeScript。第一版可先沿用 JavaScript，后续再逐步迁移 TypeScript，避免一次改动过大。
2. 后端活动管理、奖品管理等接口如果不完整，后台页面需要先使用现有数据或静态展示。
3. Stitch MCP 当前可能受网络影响无法稳定访问，UI 实现不能依赖它作为阻塞步骤。
4. PC 后台和 H5 用户端共用一个工程，后续功能增多后需要继续拆组件，避免页面文件膨胀。
